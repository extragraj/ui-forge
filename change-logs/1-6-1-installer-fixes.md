# 1.6.1 — Installer Critical Fixes

Released **2026-05-22**.

A patch release fixing four critical wiring bugs discovered while dogfooding
the new `npx ui-forge init` flow on Windows. No changes to the runtime
scripts, the prompt flow, or the lockfile schema — pure plumbing patches
behind the existing UX. Re-running `npx ui-forge init` over a 1.6.0 install
is enough to pick up every fix; legacy artifacts are migrated in place.

## Fixes

### 1. PostToolUse hook shape now matches Claude Code's actual schema (and the right env var)

**Symptom (1.6.0).** Enabling `--hooks=on` wrote a `PostToolUse` entry with
`{ id, matcher: { tool, pathGlob }, command }`. Claude Code silently rejects
that shape — it expects `{ matcher: "Edit|Write", hooks: [{ type: "command",
command: "..." }] }`. The hook never fired.

**Fix.** Rewrote `cli/src/wiring/hooks.ts` to emit the canonical shape.
The marker that lets re-installs find and replace our entry now lives on
a sibling `_uiForgeId` key (Claude Code ignores unknown keys). The cleanup
filter recognizes both the 1.6.0 legacy shape and the 1.6.1 shape, so
re-running `init` over an existing install transparently migrates the
entry — no manual cleanup needed.

Second related fix: the 1.6.0 command body referenced `$CLAUDE_FILE_PATH`,
which Claude Code does not export. The variable was never substituted, so
once the hook *did* start firing (after the shape fix above) `verify.js`
received the literal string `$CLAUDE_FILE_PATH` and fell back to its
two-arg usage error. The 1.6.1 command body uses the canonical
`$CLAUDE_TOOL_INPUT_file_path` env var (matching `references/advanced-usage.md`).
`verify.js` already short-circuits on non-TSX files and on TSX files
without the `// FORGE NOTES` header, so the bare `Edit|Write` matcher
needs no `pathGlob`.

```json
// Before (1.6.0 — broken: wrong shape + wrong env var)
{ "id": "ui-forge:verify-after-edit",
  "matcher": { "tool": ["Edit","Write"], "pathGlob": ["app/**/*.tsx"] },
  "command": "node .../verify.js \"$CLAUDE_FILE_PATH\"" }

// After (1.6.1)
{ "matcher": "Edit|Write",
  "hooks": [{ "type": "command",
              "command": "node \".../verify.js\" \"$CLAUDE_TOOL_INPUT_file_path\"" }],
  "_uiForgeId": "ui-forge:verify-after-edit" }
```

### 2. Permission entries are quoted when the install path contains spaces

**Symptom (1.6.0).** On a Windows path like `C:/Users/Garry Caber/...`,
`/forge-scan` triggered `Shell command permission check failed for pattern
"!node "C:/Users/Garry Caber/.../scan.js" "`. The wired permission was
unquoted (`Bash(node C:/Users/Garry Caber/.../scan.js:*)`) but the slash
command body invoked the script with the path quoted — Claude Code's
matcher compares strings literally, so the two never met.

**Fix.** `cli/src/wiring/permissions.ts::permissionEntries()` now emits
**both** quoted and unquoted variants when the resolved path contains
whitespace. Paths without spaces still get a single bare entry (no
visual churn for the common case). The uninstall/repair filter handles
both forms so cleanup still works.

```json
// New entries on a path with spaces:
"Bash(node \"C:/Users/Garry Caber/.../scripts/scan.js\":*)"
"Bash(node C:/Users/Garry Caber/.../scripts/scan.js:*)"
```

### 3. MCP server config uses an absolute `node` path (Cline-on-Windows fix)

**Symptom (1.6.0).** Cline (and other VS Code-embedded MCP hosts) on Windows
launch MCP servers without inheriting a shell PATH, so the bare `"command":
"node"` we wrote into `cline_mcp_settings.json` (and Codex's TOML and
Claude's `.claude.json`) failed to resolve. UI Forge MCP tools returned
"node not found"-style errors.

**Fix.** `cli/src/wiring/mcp.ts::writeMcp()` now defaults `command` to
`process.execPath` (the absolute path of the currently-running node
binary, available everywhere on every supported OS). The `mcpSnippet()`
helper and `ui-forge mcp-config` output reflect the same change. A new
optional `nodePath` parameter on `writeMcp()` lets callers override the
default if they want to encode a different node binary.

```json
// Before
{ "mcpServers": { "ui-forge": { "command": "node", "args": ["..."] } } }

// After
{ "mcpServers": { "ui-forge": {
  "command": "C:/Program Files/nodejs/node.exe",
  "args": ["..."] } } }
```

### 4. `.forgeignore` re-installs now replace the template when it's still ours

**Symptom (1.6.0).** Switching themes (e.g. `--theme=shadcn` → `--theme=
stackshift --pair=on`) and re-running `init` left the original
`.forgeignore` in place — `cli/src/wiring/forgeignore.ts` short-circuited
on any existing file, treating *every* `.forgeignore` as user-owned even
when it was the one UI Forge had just written.

**Fix.** `writeForgeignore()` now prepends a provenance header to every
generated `.forgeignore`:

```
# Generated by ui-forge@1.6.1 from references/default-stackshift-forgeignore.txt
# Delete this header to mark the file as user-owned (ui-forge will not overwrite it).
# Re-run `ui-forge repair` to regenerate after changing themes.
```

On re-install, the header is the contract: files that still carry it
are safe to replace; files without it are user-owned and preserved.
Power users can pass `--force-forgeignore` to clobber a user-owned file
explicitly.

The install summary now logs whether the file was created, replaced,
or preserved.

## New flag

| Flag | Purpose |
|---|---|
| `--force-forgeignore` | Overwrite a user-owned `.forgeignore` on re-install (1.6.0 silently preserved it, with no opt-out). |

## Tests

`tests/run-cli-tests.mjs` adds **24 new assertions** across 8 groups
covering every fix above:

- PostToolUse hook shape — string matcher, nested `hooks[]`, `_uiForgeId`
  marker, no legacy fields.
- Legacy hook entry migrates to the new shape on re-install; unrelated
  user hooks are preserved.
- Permission entries emit both quoted and unquoted variants when the
  path contains a space; bare entries when it doesn't.
- `.forgeignore` provenance header is written; re-install with a
  different theme swaps the template; user-owned files are preserved;
  `--force-forgeignore` clobbers them.
- `ui-forge mcp-config` emits an absolute node path, not bare `node`.

Total: **83 passing tests** (was 59).

## Upgrade path

1. Re-run `npx ui-forge init` (no flags needed — defaults reuse your
   existing lockfile values).
2. The legacy 1.6.0 hook entry is migrated in place; the new permission
   variants are appended; the `.forgeignore` is rewritten with a
   provenance header (idempotent — subsequent re-runs are no-ops).
3. If you customized your `.forgeignore` after the 1.6.0 install, the
   re-install will preserve it. To migrate to the new templated form,
   delete it and re-run `init`, or pass `--force-forgeignore`.

No lockfile schema change; no breaking API changes; no changes to runtime
scripts. Safe to apply via re-install.

## Not in this release (deferred)

A broader installer UX overhaul (prompt reordering, "Agentic Platforms"
rename, grouped feature multiselect, "No Theme" option, post-install
quick scan, `ui-forge.mjs` polish) is queued for **1.7.0**. Lockfile
restructure and theme-standards bridging to `design/standards/` follow
in 1.7.x. This 1.6.1 patch is scoped tightly to the four critical
wiring bugs.
